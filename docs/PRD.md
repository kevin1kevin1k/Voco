# 智慧數位溝通義肢 PRD

## 文件資訊
- 產品名稱：智慧數位溝通義肢 / Voco
- 文件版本：v0.1
- 文件狀態：初始化結構設計
- 專案階段：User Mode MVP 規劃中，Caregiver Mode 先定義資料與擴充方向

## 產品背景
Voco 是一款為高認知能力、具表達性失語症（Broca's Aphasia）的中風長者設計的 Progressive Web App（PWA）AAC 工具。目標使用者理解能力、視覺與聽覺正常，但因語言運動區受損，難以順暢說出想法。

傳統純 Grid 型 AAC 雖然容易擴充詞彙，但缺乏情境感，且常見設計偏向兒童風格，不符合長者的生活脈絡與尊嚴感。Voco 採用「寬扁型階層（Fat Hierarchy）」導航，讓使用者先從大方向概念切入，再依語境切換到最適合的呈現方式：

- `VSD（Visual Scene Display）`：以真實環境照片搭配熱點快速指認物件與意圖。
- `Grid`：以成人化、大按鈕、高辨識的方式組織較抽象或類別型詞彙。

## 目標使用者
### 主要使用者
- 中風後具表達性失語症的長者
- 保有生活語意理解能力與判斷力
- 需要低挫折、低學習成本、可快速表達需求的輔具

### 次要使用者
- 家人與照護者
- 未來需要透過照護者模式管理詞彙、照片、熱點與使用紀錄

## 產品目標
1. 提供可離線使用、可安裝到主畫面的 AAC 工具。
2. 以 Fat Hierarchy 降低瀏覽大量詞彙時的迷失感。
3. 以 VSD + Grid 混合介面同時滿足情境表達與分類組織需求。
4. 提供語音輸出，優先使用 `zh-TW` 台灣口音。
5. 保持成人化、尊重式、可讀性高的無障礙介面。
6. 以本地互動紀錄支援輕量推薦，逐步改善常用詞彙可達性。

## 非目標
- 首版不建置後端服務與帳號系統。
- 首版不提供雲端同步。
- 首版不實作真正串接的 Vision API，自動熱點生成僅保留規格與擴充方向。
- 首版不導入複雜機器學習模型，推薦邏輯以本地規則為主。

## 核心使用情境
### 情境一：在住家場景中快速表達需求
1. 使用者從首頁點擊「住家」。
2. 系統載入住家實景照片。
3. 使用者點擊照片上的熱點，例如電視、沙發、窗戶。
4. 系統立刻朗讀對應句子，例如「我想看電視」。

### 情境二：在家人分類中選擇稱謂
1. 使用者從首頁點擊「家人」。
2. 系統切換為成人化 Grid。
3. 使用者點擊「兒子」、「女兒」等稱謂。
4. 系統根據近期使用歷史，在推薦區顯示相關但未必最常按的詞彙。

### 情境三：照護者未來擴充內容
1. 照護者上傳實景照片。
2. 系統未來可依 Vision API 生成候選熱點與語音文案。
3. 照護者人工調整後發布到使用者模式。

## 功能需求
### 1. 階層式導航
- 首頁需顯示大分類，例如住家、地點、家人、醫療。
- 主導航需固定在畫面上方或固定區域，讓使用者能快速切換分類。
- 需支援返回上一頁與回首頁。
- 需清楚標示目前所在分類或板面。

### 2. 動態視圖渲染
- 系統依 board 資料中的 `ext_voco_display_type` 決定使用 `VSD` 或 `Grid`。
- `VSD` 板面需支援背景圖、熱點定位與可點擊發聲。
- `Grid` 板面需支援按鈕圖片、符號、文字與語音輸出。
- 所有板面需透過同一套 board 資料模型驅動，不拆成兩種資料來源。

### 3. 語音輸出
- 使用 `window.speechSynthesis`。
- 預設語系為 `zh-TW`。
- 優先選擇 `zh-TW` voice；若裝置無可用 voice，需有清楚 fallback 行為。
- 發聲內容優先使用 `vocalization`，沒有則退回 `label`。

### 4. 輕量推薦
- 僅在 Grid 模式啟用。
- 推薦來源以本地點擊紀錄與類別關聯為主。
- 推薦區需與固定詞彙區分，避免混淆。
- 推薦結果需排除不可用或無意義的重複項。

### 5. Caregiver Mode
- 首版先保留資料需求與擴充介面，不要求完整 UI。
- 至少需要有可儲存與分析使用紀錄的前端狀態。
- 未來 AI 建置精靈需以照片、候選熱點、語音文案為主要輸入輸出。

## 無障礙與 UX 原則
- 介面文字與按鈕需符合長者閱讀與點擊需求。
- 主要互動元件需有可見 focus 狀態，支援鍵盤操作。
- 顏色對比需足以辨識，不依賴顏色作為唯一資訊來源。
- VSD 熱點需兼顧可點擊性與場景可讀性，不可完全隱形到失去操作線索。
- 視覺語言需避免幼稚化，保留成人生活情境與尊嚴感。
- 操作結果需即時回饋，並避免誤觸造成高挫折。

## 技術與資料契約
### 前端技術
- React + Vite
- Redux Toolkit（RTK）
- PWA：`vite-plugin-pwa`
- Web Speech API：`window.speechSynthesis`

### 資料格式
- 核心資料標準遵循 Open Board Format（OBF）JSON。
- Voco 擴充欄位一律使用 `ext_voco_` 前綴。

### 目前定義中的擴充欄位
- `ext_voco_display_type`
  - 值：`grid` 或 `vsd`
  - 用途：決定板面的渲染模式
- `ext_voco_background`
  - 欄位：`image_id`、`width`、`height`
  - 用途：VSD 場景背景資訊
- `ext_voco_hotspot`
  - 欄位：`x`、`y`、`width`、`height`、`shape`
  - 用途：定義 VSD 熱點的相對座標與形狀

### 導航與資料流
- 板面透過 `load_board.id` 串接下一層板面。
- 導航狀態需至少包含：
  - `currentBoardId`
  - `history`
  - `rootBoardId`
- 使用紀錄需至少包含：
  - `buttonId`
  - `boardId`
  - `timestamp`

## 目前專案現況

- 已有 React + Redux Toolkit 專案基底。
- 已有 `root`、`home`、`family`、`medical`、`places` OBF 板面資料。
- 已有 `BoardRenderer`，可依 `ext_voco_display_type` 在 `Grid` 與 `VSD` 間切換。
- 已有階層式導航 slice。
- 已有 Web Speech API hook 與語音設定 slice。
- 已有推薦 slice 與點擊紀錄基礎。
- 已設定 PWA build，可產出 service worker。
- 已補上基線 PWA icon 與住家 mock VSD 場景資產。

### 已知落差
- README 仍是預設 Vite 內容。
- 推薦邏輯目前偏簡化，尚未充分反映語意關聯。
- Caregiver Mode 已有編輯雛形，但資訊架構與正式操作流程仍未定案。
- 尚未建立正式測試規格、自動化驗證流程與 README 專案化說明。

## User Mode MVP 驗收標準
1. 使用者可從首頁穩定進入現有板面，且導航不會出現死路。
2. `住家` 可正確顯示 VSD，熱點可發聲。
3. `家人` 可正確顯示 Grid，按鈕可發聲。
4. Grid 推薦區會根據點擊歷史更新，且不顯示無效項目。
5. App 可 build、可安裝、可在離線情況下啟動基本介面。
6. 主要操作具備基礎可存取性，包括鍵盤焦點、ARIA 標記與足夠點擊區。
